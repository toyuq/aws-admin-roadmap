# Step 05. 비용 관리

## 목표

- 비용이 어디서 발생하는지 파악한다.
- 불필요한 지출을 줄이는 습관을 만든다.

## 복습용 정리

### 핵심 AWS 비용 관리 서비스 한 줄 요약

- **AWS Cost Explorer**: 서비스/계정/태그 기준으로 비용 추이를 시각화하고 비교 분석하는 도구
- **AWS Budgets**: 예산 한도를 설정하고 임계치(예: 80%, 100%) 도달 시 알림을 보내는 서비스
- **AWS Cost and Usage Report (CUR)**: 가장 상세한 비용/사용량 원본 데이터(시간 단위 수준) 보고서
- **AWS Cost Anomaly Detection**: 평소 패턴 대비 비정상 비용 증가를 자동 감지하는 서비스
- **AWS Pricing Calculator**: 아키텍처 설계 전에 예상 비용을 계산하는 도구
- **Trusted Advisor (비용 최적화 항목)**: 유휴 리소스, 저활용 리소스 등 절감 포인트를 제안
- **Compute Optimizer**: EC2/EBS/Lambda 리소스 권장 스펙을 제시해 과다 프로비저닝 비용 절감
- **Savings Plans / Reserved Instances**: 사용량 약정을 통해 온디맨드 대비 단가를 낮추는 할인 모델
- **AWS Backup**: 백업 주기/보존 정책을 통합 관리하여 불필요한 백업 중복 비용을 줄이는 서비스

### 비용 최적화에서 자주 같이 쓰는 조합

- **Cost Explorer + CUR**: 요약 그래프와 원본 상세 데이터를 함께 분석
- **Budgets + SNS/Email**: 예산 임계치 기반 실시간 알림 체계 구축
- **Cost Anomaly Detection + 담당자 알림 채널**: 급격한 비용 증가를 조기 탐지
- **태그 정책 + Cost Allocation Tags**: 팀/프로젝트/환경별 청구 분리
- **Trusted Advisor + Compute Optimizer**: 유휴 자원 제거 + 스펙 다운사이징

### 비용 관리에서 알아둘 핵심 개념

- **고정비 vs 변동비**: RI/SP 같은 약정형 비용과 사용량 기반 비용을 분리해서 본다.
- **단가(Price) vs 사용량(Usage)**: 비용은 단가 × 사용량이므로, 둘 중 어느 쪽이 증가했는지 먼저 분리 분석한다.
- **데이터 전송 비용**: 같은 리전 내부/다른 AZ/인터넷 아웃바운드에 따라 비용이 크게 달라진다.
- **스토리지 라이프사이클**: 자주 안 쓰는 데이터는 저렴한 스토리지 클래스로 자동 이동한다.
- **백업 보존 기간**: 보존 기간이 길수록 비용이 누적되므로 복구 목표(RPO/RTO)에 맞춰 설계한다.

## 학습 내용

- 태그 전략
- 예산 알람
- 저장소 비용
- 트래픽 비용
- 스냅샷 / 백업 비용

### 태그 전략 (Cost Allocation Tags)

- **필수 태그 기준 예시**: `Project`, `Owner`, `Environment`, `CostCenter`, `Service`
- 리소스 생성 시 태그 누락을 줄이기 위해 IaC(CloudFormation/Terraform) 기본 태그를 강제한다.
- 태그 표기 규칙(예: `prod`, `dev`, `stg`)을 통일하지 않으면 비용 집계가 분산되어 분석 품질이 떨어진다.
- 월말 정산 시 태그 미부착 리소스를 별도 목록으로 추출해 보완한다.

### 예산 알람 (AWS Budgets)

- 월간 총예산 + 서비스별 예산을 분리 설정한다. (예: 전체 50만 원, EC2 20만 원)
- 임계치 예시: 50% (주의), 80% (경고), 100% (긴급)
- 실제 운영에서는 **Forecasted(예상 초과)** 기준 알람을 함께 켜야 선제 대응이 가능하다.
- 예산 초과 시 즉시 확인할 항목:
	- 최근 7일 서비스별 증감
	- 신규 생성 리소스 여부
	- 데이터 전송량 급증 여부

### 저장소 비용 (S3/EBS/EFS/RDS)

- **S3**: Standard, Standard-IA, One Zone-IA, Glacier 계열로 접근 빈도에 맞춰 클래스 최적화
- **S3 Lifecycle**: 일정 기간 이후 자동 전환/삭제로 장기 보관 비용 관리
- **EBS**: 미사용 볼륨/스냅샷 정리, 과도한 Provisioned IOPS 여부 점검
- **EFS**: 스토리지 클래스 전환(IA) 및 접근 패턴 기반 비용 최적화
- **RDS 스토리지**: 필요 이상 고성능 옵션/프로비저닝 크기 사용 여부 점검

### 트래픽 비용 (Data Transfer)

- 같은 리전이라도 AZ 간 트래픽 비용이 발생할 수 있으므로 아키텍처 배치를 점검한다.
- 인터넷 아웃바운드 비용은 누적 영향이 크므로 CDN(CloudFront) 활용을 우선 검토한다.
- NAT Gateway는 편하지만 트래픽이 많으면 비용이 커질 수 있어 사용 패턴을 모니터링한다.
- 로그/백업/복제 트래픽도 숨은 비용이므로 주기적으로 상위 트래픽 원인을 확인한다.

### 스냅샷 / 백업 비용

- 스냅샷은 "한 번 찍고 끝"이 아니라 누적 저장량이 비용을 만든다.
- 백업 정책은 시스템 중요도별로 차등 적용한다.
	- 핵심 서비스: 짧은 백업 주기 + 적절한 장기 보관
	- 비핵심 서비스: 주기 완화 + 빠른 만료 정책
- 복구 테스트를 하지 않으면 백업은 비용만 쓰고 효과가 불확실할 수 있다.
- 중복 백업(서비스 자체 백업 + 별도 백업 도구) 여부를 확인해 이중 비용을 줄인다.

### 월간 비용 점검 루틴 (실무형)

- 1주차: 지난달 비용 Top 5 서비스와 증가율 확인
- 2주차: 태그 누락 리소스 정리 + 미사용 리소스 제거
- 3주차: Savings Plans/RI 적용률 검토
- 4주차: 다음 달 예산 및 알람 임계치 재조정

### 이 단계에서 스스로 답해볼 질문

- 이번 달 비용 증가의 주원인은 단가 상승인가, 사용량 증가인가?
- 태그 누락 리소스가 어느 팀/프로젝트 비용으로 섞여 있는가?
- 데이터 전송 비용이 높은 경로는 어디인가?
- 백업 보존 기간은 실제 복구 목표(RPO/RTO)에 맞는가?
- 온디맨드 사용 중 약정(Savings Plans/RI) 전환 가능한 워크로드는 무엇인가?

### 암기용 짧은 정리

- Cost Explorer = 비용 시각화/비교
- Budgets = 예산 임계치 알림
- CUR = 상세 원본 비용 데이터
- Cost Anomaly Detection = 이상 비용 자동 탐지
- 태그 전략 = 비용 귀속의 시작점
- Lifecycle = 스토리지 장기비용 절감 핵심
- 데이터 전송 = 숨은 고비용 구간

## 기록

- 오늘 배운 내용:
- Cost Explorer/CUR/Budgets를 함께 써야 "보는 비용"에서 "통제하는 비용"으로 전환된다.
- 비용 절감은 단순 삭제보다 태그 체계와 예산 알람 자동화가 먼저다.
- 스토리지와 트래픽은 작은 누수가 누적되어 월말에 큰 차이를 만든다.

- 비용 증가 원인:
- 태그 누락으로 비용 귀속이 불명확해 추적이 지연됨
- NAT Gateway/인터넷 아웃바운드 트래픽 급증
- 장기 미삭제 스냅샷, 저접근 데이터의 고비용 스토리지 클래스 유지

- 절감 가능 항목:
- 태그 표준화 + 미부착 리소스 점검 자동화
- S3 Lifecycle 적용 및 불필요 스냅샷 정리
- 데이터 전송 경로 최적화(CloudFront, AZ 간 트래픽 최소화)
- 저활용 EC2 다운사이징 또는 Savings Plans/RI 검토

- 다음 점검 때 확인할 내용:
- 예산 Forecasted 알람이 적절히 동작하는지
- 상위 3개 비용 서비스의 전월 대비 증감률
- 백업 보존 기간과 실제 복구 요구사항의 정합성

- [ ] 정리 가능한 리소스를 찾았다
- [ ] 태그가 잘 붙어 있는지 점검했다
- [ ] 데이터 전송 비용 상위 경로를 파악했다
- [ ] 스냅샷/백업 보존 정책을 검토했다
